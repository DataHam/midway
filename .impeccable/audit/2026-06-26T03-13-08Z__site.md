## Detailed Findings by Severity

### [P0] helen.html has no stylesheet — page renders unstyled
- **Location**: `helen.html` `<head>` (no `cdn.tailwindcss.com` script, no `<link rel="stylesheet">`). Contrast with `index.html:39` and `danny.html:36`, which both load the CDN.
- **Category**: Theming / Blocking
- **Impact**: The page a visitor reaches when they tap "Meet Helen" is broken — unreadable hierarchy, giant icons, default link styling. It directly defeats the site's one job (route to the right person, then introduce them).
- **Recommendation**: Add the identical `<script src="https://cdn.tailwindcss.com"></script>` + `tailwind.config` block that `danny.html` carries (lines 36 & 68–~120). Long-term, both pages should move to the compiled `public/css/styles.css` (the deferred `/impeccable optimize` step) so production doesn't depend on the CDN at all.
- **Suggested command**: `/impeccable optimize` (or a direct fix — it's a copy/paste of danny.html's head).

### [P1] Brand font missing from repository
- **Location**: `assets/fonts/Montserrat/` (only `README.md` present); `@font-face` blocks in all three HTML files reference `Montserrat-{Regular,SemiBold,Bold}.woff2`.
- **Category**: Theming / Performance
- **Impact**: Typography identity ("Weight-Is-Hierarchy" from DESIGN.md) doesn't render; pages fall back to system fonts. Three 404s per page load.
- **WCAG/Standard**: n/a (brand integrity, not a11y).
- **Recommendation**: Commit the three `.woff2` files (or restore from the font source), or self-host via a build step. Verify the `crossorigin` preloads then resolve.
- **Suggested command**: `/impeccable typeset` (after fonts are restored) to confirm the loading strategy.

### [P1] Profile photos — unequal visibility & clashing backgrounds
- **Location**: `assets/images/optimized/HelenPortrait-*.webp`, `DannyPortrait-*.webp`; rendered in the index.html diptych; source PNGs in `Downloads/`.
- **Category**: Theming / Responsive (visual consistency)
- **Measured**:
  - **Subject scale**: Helen is framed *waist-up* with generous headroom — her head fills ~20% of the frame. Danny is framed *chest-up*, tighter — his head fills ~28–30%. In the equal-size circular crops they sit in, Helen reads small and distant; Danny reads close and present.
  - **Background**: Helen's backdrop sampled `rgb(47,71,65)` — a **dark** desaturated forest green (low luminance). Danny's sampled `rgb(255,255,255)` / `rgb(182,173,172)` — **bright** white/warm-grey, with a dark brown window edge (`rgb(68,30,5)`) intruding bottom-right. Opposite luminance *and* opposite temperature.
  - On the `#103248` navy zone, Helen's dark-green disc nearly merges into the ground while Danny's bright disc pops — so the visual weight is lopsided toward Danny.
- **Impact**: Undercuts the equal-footing premise of the router. The site's whole purpose is to present the two people as parallel choices; the imagery makes Helen secondary.
- **Recommendation** — two parts:
  1. **Equalize framing.** Re-crop Helen ~1.3× tighter so her head/shoulders match Danny's. Target a shared spec for both circles: head-top ~15–16% from the top, eyeline ~38–40%, shoulders filling the base. Re-export 375/768 webp from the tighter crop. (Danny is already close; nudge if needed.)
  2. **Unify the backgrounds to one tone.** Because *both* subjects wear dark/navy clothing, a dark backdrop would flatten them into silhouettes — so go **light**. Recommended: a single cool neutral, mist-grey **`#F1F3F4`** (an existing DESIGN.md token), giving two bright, equal discs that lift both faces off the navy and let the teal hover-ring read. Brand-flavoured alternative: a pale desaturated coastal-teal (~`#DCE8E5`) to echo the ring and carry "warmth" without a cream default. **Do not** match the disc to navy (silhouette merge) and **do not** reach for cream/beige (the AI warm-neutral default). A soft radial (slightly lighter behind the head) reads more premium than a flat fill if you want the extra polish.
- **Suggested command**: `/impeccable polish` (or a direct re-export with sharp + matte/recolor).

### [P2] Desktop hero crop subordinates Helen
- **Location**: `index.html` `.hero-photo { object-position: 50% 22% }` (line ~119/134).
- **Category**: Responsive / composition
- **Impact**: On the wide desktop band the family photo crops so Danny sits center while Helen is pushed to the left edge and partially cut — the same "Helen is secondary" pattern as the portraits. (Mobile crops both faces well.)
- **Recommendation**: Shift `object-position-x` toward Helen on desktop (try `42% 22%`), or add an `@media (min-width:768px)` crop tuned so both faces are balanced. Re-check at 1024/1280/1440.
- **Suggested command**: `/impeccable adapt`.

### [P2] Hero image paints late (perceived blank hero)
- **Location**: `index.html` hero `<img>` (preloaded `DHFamily-1280.webp`).
- **Category**: Performance
- **Impact**: First paint can show an empty navy band before the large webp decodes (observed on first load; console warns "preloaded but not used within a few seconds"). The hero is `loading="eager"`/`fetchpriority="high"` already, so this is decode latency on a ~1923px-tall image, not a fetch miss.
- **Recommendation**: Add a low-cost LQIP/blur placeholder or a navy→photo cross-fade so the band is never empty; consider a smaller default `src` for the hero breakpoint. Verify the `<img>` `width/height` match the asset to reserve space (it does — no CLS observed).
- **Suggested command**: `/impeccable optimize`.

### [P2] Person names are not headings on index.html
- **Location**: `index.html` diptych — "Helen Tam-Tham" / "Danny Tam-Tham" are not `<h2>`; the page exposes only one `<h1>` ("Good to meet you.").
- **Category**: Accessibility (semantics) / SEO
- **Impact**: Screen-reader users navigating by heading, and crawlers, don't get the two names as document structure — weak for two pages that are literally about two named people.
- **Recommendation**: Promote each name to `<h2>` inside its zone link (keep the existing descriptive `aria-label` on the `<a>`).
- **Suggested command**: `/impeccable clarify` or direct.

### [P3] Favicon PNG variants missing
- **Location**: All pages reference `/assets/images/favicons/favicon-16x16.png`, `-32x32.png`, `apple-touch-icon.png`; only `assets/images/favicons/favicon.ico` exists. Two/three 404s per load.
- **Recommendation**: Generate the PNG variants from the source mark, or drop the `<link>`s you won't ship.

### [P3] Both portraits `loading="eager"`
- **Location**: `index.html` portraits — both eager. On mobile Danny's portrait is well below the fold.
- **Recommendation**: Set Danny's portrait `loading="lazy"` (and Helen's too on mobile, where it's also below the hero). Minor.

## Patterns & Systemic Issues
- **Helen is systematically subordinated to Danny by the imagery** — broken page (P0), smaller/recessive portrait (P1), and the Danny-centered hero crop (P2) all push the same direction. For a site whose entire job is presenting the two as equal choices, this is the most important *theme*, not just three separate bugs. Fixing the three together restores parity.
- **Inconsistent styling pipeline across pages** — index & danny use the Tailwind CDN, helen uses nothing, and a compiled `public/css/styles.css` exists but is linked by none of them. Converging all three on the compiled stylesheet (the deferred optimize step) removes both the CDN production-warning and the helen.html breakage class of bug.

### CSS convergence (the `/impeccable optimize` step — done)
Moved all three pages off the Tailwind **Play CDN** onto the project's compiled stylesheet.
- The project is **Tailwind v4** (`@import "tailwindcss"` + `@theme`). The `@theme` color tokens were named `--color-navy`/`--color-yellow`/… but every page uses `primary-navy`/`secondary-teal`/… utilities (from the old inline CDN config) — a mismatch that would have nulled every brand color on compile. Renamed the tokens to `--color-primary-navy`, `--color-primary-yellow`, `--color-secondary-{teal,red,blue,grey}` so the compiled utilities match the markup, and added `@source "../../*.html"` for v4 source detection.
- Replaced `<script src="cdn.tailwindcss.com">` + the inline `tailwind.config` block in `index.html`, `danny.html`, `helen.html` with `<link rel="stylesheet" href="/css/styles.css">`.
- Compiled `public/css/styles.css` (≈80 KB) via the project's `postcss` pipeline; the production `npm run build` regenerates it. Verified all three pages render identically off the compiled CSS (served from a `public/` preview): `window.tailwind` now `undefined`, body navy `rgb(16,50,72)`, role text teal `rgb(125,194,182)`, Montserrat loaded, 0 console errors, **no more CDN production warning**.
